# ast-grep Rule Workflows

## Decision Flow

1. Use `sg run` for quick single-pattern matching and ad-hoc rewrites.
2. Use `sg scan` for YAML rules with relational and composite logic.
3. Test on small examples before scanning entire repositories.

## Core Commands

```bash
# One-off structural search
sg run -p '<pattern>' -l <language> <paths...>

# One-off structural rewrite
sg run -p '<pattern>' -r '<rewrite>' -l <language> -i <paths...>

# Rule-based scan from sgconfig.yml
sg scan <paths...>

# Rule-based scan with a single file
sg scan --rule rules/<rule-id>.yml <paths...>

# Run rule tests
sg test
```

## Rule Construction Progression

1. Start with `pattern`.
2. Add `kind` when node type needs tightening.
3. Add relational logic (`has`, `inside`) for parent-child constraints.
4. Add composite logic (`all`, `any`, `not`) for complex conditions.
5. For relational rules, default to `stopBy: end`.

Example relational rule:

```yaml
id: async-with-await
language: JavaScript
rule:
  kind: function_declaration
  has:
    pattern: await $EXPR
    stopBy: end
```

## Inline Rule Shell Escaping

In double-quoted shell strings, escape metavariables to avoid shell substitution.

```bash
# Good: escaped metavariable in double quotes
sg scan --inline-rules "id: t
language: JavaScript
rule:
  pattern: console.log(\\$ARG)" .

# Also good: single-quoted rule text
sg scan --inline-rules 'id: t
language: JavaScript
rule:
  pattern: console.log($ARG)' .
```

## Validation Sequence

1. Test against `--stdin` for fast iteration.
2. Test rule files against dedicated example files.
3. Inspect parse/debug output with `--debug-query`.
4. Run repository scan once matches are stable.

```bash
echo "async function x(){ await fetch(); }" | sg scan --inline-rules 'id: t
language: JavaScript
rule:
  kind: function_declaration
  has:
    pattern: await $EXPR
    stopBy: end' --stdin
```

## JSON Output Modes

```bash
# One JSON object per line for pipelines
sg run -p 'console.log($$$ARGS)' -l TypeScript --json=stream .

# Compact JSON array for artifacts
sg scan --json=compact --include-metadata .
```

## Generated Starter Files

The following are generated by current `sg new` commands.

### `sg new project -y`

```yaml
ruleDirs:
- /absolute/path/to/rules
testConfigs:
- testDir: /absolute/path/to/rule-tests
utilDirs:
- /absolute/path/to/utils
```

### `sg new rule <id> -l <Language> -y`

```yaml
# yaml-language-server: $schema=https://raw.githubusercontent.com/ast-grep/ast-grep/main/schemas/rule.json

id: no-console-log
message: Add your rule message here....
severity: error # error, warning, info, hint
language: TypeScript
rule:
  pattern: Your Rule Pattern here...
# utils: Extract repeated rule as local utility here.
# note: Add detailed explanation for the rule.
```

### `sg new test <id> -y`

```yaml
id: no-console-log
valid:
- "valid code"
invalid:
- "invalid code"
```

## Safe Rewrite Sequence

1. Search only with `sg run -p`.
2. Rewrite interactively with `-i`.
3. Apply all with `-U` only after review.
4. Re-run `sg test`, language tests, lint, and typecheck.
